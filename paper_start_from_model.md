## 5. Models and Technologies

### 5.1 Model Usage and Design Choices
The system is designed as a hybrid multi-stage financial reasoning framework, in which structured computation and LLM-based reasoning are assigned to different parts of the pipeline according to their functional roles. In the current design, the analyst layer is primarily responsible for extracting structured evidence from different data modalities, while the debate and portfolio stages are responsible for synthesis, argumentation, comparison, and final judgment, where LLM-based reasoning is more appropriate. The final risk management stage remains largely rule-based, so that portfolio constraints and allocation adjustments are applied in a more deterministic and controllable manner. This division of labor helps reduce unnecessary generative freedom in early-stage analysis, which is especially important in financial contexts, where factual consistency and comparability across different data modalities are critical.

#### Analyst Team
The analyst stage consists of four specialized modules for fundamental, technical, macro, and news & trend analysis. It allows four distinct analyst modules to process feature-based inputs from different sources, outputting standardized JSON analysis reports with uniform formats and fields. These reports include signals, confidence scores, summaries, bullish and bearish factors, risk flags, and key metrics used in the analysis, which makes the analyst layer easier to integrate with subsequent stages.

#### Debate & Portfolio Judge
A custom Python LLM client that supports multiple inference providers, including Groq and Anthropic, is used in the debate and portfolio stages. In the current implementation, debate-stage reasoning uses a shared LLM client with provider and model selected by configuration, with Groq-hosted Llama supported as one option, whereas portfolio-level judgment uses a Claude model through Anthropic. LLM calls are further constrained by low-temperature settings and JSON-oriented prompting, which help keep the generated outputs structured and reduce unnecessary variability in downstream processing.

These stages are also designed with deterministic fallback behavior. In the debate stage, both case construction and final judgment attempt an LLM call first, but switch to rule-based logic if model calls fail. The portfolio judge follows a similar pattern: it uses a single LLM call to rank all tickers jointly and assign relative weights, but falls back to a confidence-weighted allocation among bullish tickers when needed.

Another implementation choice is the use of lightweight memory in the debate stage. The system maintains structured local memory through JSON artifacts, including both short-term debate context and cross-week ticker-level state, which can be referenced by the bull, bear, and judge agents as background information. The prompts place explicit constraints on how memory is used. It can support continuity across rounds and weeks, but it cannot override current analyst evidence or be used to introduce new current-week facts. The orchestration layer is implemented through custom Python scripts rather than external agent frameworks such as LangChain or LlamaIndex. The current implementation does not rely on embedding models, vector databases, or retrieval-augmented generation, and the debate memory mechanism is implemented through local JSON state rather than embedding-based retrieval.

#### Risk Management
The final risk management stage is primarily rule-based. After the portfolio judge produces weekly weights, the risk manager applies deterministic portfolio-level constraints, including single-position caps, sector caps, minimum-holdings requirements, defensive equal-weight mode, and iterative redistribution when caps are violated. Only after these adjustments are applied does the module call an LLM to generate a risk commentary over the adjusted allocation and debate highlights.

Overall, the current implementation emphasizes structured intermediate representations, constrained prompting, stage-specific model usage, and rule-based safeguards. This combination is consistent with the overall design goal of building a multi-stage financial reasoning system that remains modular, interpretable, and operationally robust.

### 5.2 Infrastructure and Deployment Considerations
From an engineering perspective, the system is implemented as a script-driven and batch-oriented research pipeline that depends on several external services and data sources, including Yahoo Finance, SEC EDGAR, FRED, Finnhub, and Google Trends. The system prioritizes modular experimentation over deployment-oriented complexity. Dependency management is handled through a pyproject.toml-based Python project setup with Hatchling build configuration and optional development dependencies, environment variables are used for API and model configuration, and both raw inputs and intermediate artifacts are stored in parquet, CSV, and JSON formats. The pipeline is also explicitly time-aware, since historical reports are generated according to reporting frequency and date alignment rules, which support backtesting consistency and help reduce information leakage. Overall, the technical stack is designed in a way that emphasizes modularity, reproducibility, and controlled experimentation, which makes it more suitable for a research prototype than for a production deployment setting.


## 6. Responsible AI Considerations
### 6.1 Bias and fairness analysis
Concerns regarding bias and fairness in the current system primarily relate to market coverage, data scope, and model behavior. First, the system is evaluated on a fixed sample of six large-cap U.S. stocks. Therefore, the resulting assessments and conclusions cannot be directly generalized to the broader stock market, and are more likely to reflect the behavior of companies with high liquidity, high media visibility, and large market capitalizations. Second, the system relies on a limited number of external data providers, which may introduce source-specific coverage bias or result in inconsistent information quality across different companies. Third, the system does not incorporate the full range of historical data prior to the testing period, which may lead it to understate the influence of major earlier market events on current output results. Finally, if the supplied evidence does not fully override the model's prior knowledge regarding well-known companies, such prior knowledge may still influence its judgments.

### 6.2 Hallucination and Factual Accuracy Controls
To mitigate the risk of hallucinations and improve factual accuracy, the system employs several practical safeguards. First, structured JSON outputs are used in major analytical and reasoning stages, rather than unrestricted free-form text. Additionally, before processing by a large language model (LLM), the analyst layer extracts structured evidence primarily through rule-driven methods. During the debate and judgment phases, prompts explicitly instruct the model to use only the provided inputs and prohibit the fabrication of facts or information. Furthermore, fallback logic helps maintain usable and bounded system behavior in the event of parsing failures or when model calls are unavailable. It should be noted, however, that while these measures reduce the generation of unsubstantiated content, they do not completely eliminate the risk of hallucinations.

### 6.3 Privacy considerations
Privacy concerns with this system are relatively limited, as it primarily uses publicly available market, financial, macroeconomic, news, and search-interest data rather than individual user data. However, external LLM calls and stored transcripts or rationales still introduce limited data-governance and security considerations, so the current implementation should be viewed as a research prototype rather than a privacy-hardened production system.

### 6.4 Safety measures and ethical implications
Safety and ethics are especially important because users may overtrust model-generated rationales, as the system produces signals, confidence scores, rankings, and portfolio allocations that may appear authoritative. To reduce this risk, the risk management layer applies deterministic constraints such as single-position caps, minimum-holdings requirements, defensive equal-weight mode, and allocation normalization before generating any LLM-based commentary. The project also remains within a paper-trading and backtesting setting rather than direct broker execution, which limits real-world operational risk.

## 7. Findings and Discussion

## 8. Conclusion and Future Work
In summary, this project develops a hybrid multi-stage framework for stock analysis and portfolio construction that combines structured analyst modules, adversarial debate, portfolio-level judgment, risk management, and backtesting within a single pipeline. Its main contribution is not merely the use of LLMs, but the integration of structured evidence extraction, debate-based comparison, and rule-based safeguards into one coherent system. Through this design, the project provides a prototype approach to combining modular analysis, comparative decision-making, and controlled portfolio construction in a research-oriented setting.

The current implementation also has several important limitations. The system is evaluated on a fixed universe of six large-cap U.S. stocks, so its findings should not be treated as directly representative of the broader market. In addition, the debate memory mechanism remains lightweight and local, rather than a more systematic long-horizon memory design. The system also continues to rely on limited external data sources and does not include a dedicated factual-verification layer beyond prompt constraints, structured outputs, and fallback logic. Finally, the backtesting setup remains simplified and does not incorporate transaction costs, slippage, taxes, or live execution frictions, which limits the system's real-world applicability.

The future directions for improving this project are relatively clear. One is to expand the stock universe to include more companies, industries, and market-cap levels, so that the system can be tested under broader market coverage and more varied information conditions. Another direction is to evaluate the debate process more systematically, including the quality and consistency of bull and bear arguments, as well as their actual contribution to final portfolio decisions. In addition, it would be valuable to introduce more realistic investment constraints in the portfolio and risk stages, including transaction costs, tax assumptions, etc.
